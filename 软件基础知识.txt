# 软件基础知识自证

## 1. 需求工程

**面向对象分析（OOA）**：掌握从问题域中识别对象、属性、行为和关系的分析方法；理解 OOA 的核心产出——领域模型（概念类图），通过抽象现实世界实体建立系统的概念框架。

**Use Case（用例分析）**：掌握用例图建模——识别参与者（Actor）和系统用例，描述系统与外部交互的功能边界；熟悉用例规约的编写（前置条件、主成功场景、扩展场景、后置条件）；理解用例驱动开发在需求捕获与沟通中的实践价值。

**实例化需求分析**：掌握通过具体实例（Example）驱动需求澄清的方法——用真实的输入输出样例替代抽象描述，消除需求二义性；理解 BDD（行为驱动开发）中 Given-When-Then 格式将实例化需求转化为可执行规约的实践。

## 2. 设计原则

**结构化设计**：掌握自顶向下、逐步求精的模块化设计方法；理解高内聚低耦合原则——模块内部功能紧密相关（高内聚），模块间依赖最小化（低耦合）；熟悉数据流图（DFD）作为结构化分析的核心建模工具。

**面向对象设计（OOD）**：掌握 SOLID 五大设计原则——单一职责（一个类只有一个变化原因）、开闭原则（对扩展开放对修改关闭）、里氏替换（子类可替换父类不破坏行为）、接口隔离（接口细粒度拆分避免臃肿依赖）、依赖倒转（高层不依赖低层，都依赖抽象）；理解这些原则在实际代码设计中指导可维护性和扩展性的具体作用。

**设计模式**：熟练掌握常用设计模式及其应用场景——
- 创建型：工厂方法（解耦对象创建）、单例（全局唯一实例）、建造者（复杂对象分步构建）
- 结构型：代理模式（控制对象访问，如远程代理/缓存代理）、适配器（接口转换兼容）、装饰器（动态扩展功能）
- 行为型：观察者模式（一对多事件通知，发布-订阅解耦）、策略模式（运行时切换算法，消除条件分支）、模板方法（定义算法骨架，子类实现具体步骤）

具备在项目中灵活运用观察者、策略、代理等设计模式解决实际工程问题的实践经验。

**华为面向对象设计原则**：熟悉华为公司在 OOD 实践中强调的设计规范——模块职责清晰、接口契约明确、防御性编程、代码可测试性设计等工程化要求。

## 3. 编程语言

**Python**：熟练掌握 Python 作为 AI/ML 主力语言的工程应用——数据处理（Pandas/NumPy）、模型开发（PyTorch/TensorFlow）、脚本自动化；掌握 Python 高级特性（装饰器、生成器、上下文管理器、元类）；理解 GIL 对多线程的限制及多进程/异步 IO 的解决方案。

**Shell**：掌握 Shell 脚本在运维自动化、数据处理 pipeline、任务调度中的应用；熟悉常用命令组合（管道、重定向、awk/sed/grep 文本处理）。

**Java**：熟练掌握 Java 面向对象编程、集合框架、多线程并发（synchronized/Lock/线程池）、JVM 内存模型与垃圾回收机制；已获得 Java 软件可信能力专业级证书；具备组件式开发思维与能力。

**C/C++**：掌握 C/C++ 在高性能计算和底层系统开发中的应用；理解内存管理（栈/堆、指针、RAII）、模板编程、STL 容器与算法；了解 C++ 在推理引擎和算子开发中的核心地位。

**Scala**：掌握 Scala 函数式+面向对象混合编程范式；熟悉 Scala 在 Spark 生态中的应用——Spark 核心 API 基于 Scala 编写，用 Scala 开发 Spark 应用具有原生优势。

## 4. 机器学习框架

**MindSpore**：掌握华为全场景 AI 框架的核心特性——自动微分（基于源码转换的 AD）、自动并行策略、图算融合优化；熟悉 MindSpore 在昇腾 NPU 上的原生适配。

**TensorFlow**：掌握 TF 的静态图（1.x）到动态图（Eager Execution/2.x）的演进；熟悉 Keras 高层 API 快速原型开发、tf.data 数据 pipeline 构建、SavedModel 模型导出流程。

**PyTorch**：熟练掌握 PyTorch 动态图（define-by-run）的核心优势——灵活调试、直观的 Python 控制流；掌握 Autograd 自动微分机制、DataLoader 数据加载、nn.Module 模型定义范式；熟悉 TorchScript/torch.compile 对部署和性能优化的支持。

**MXNet**：了解 MXNet 混合前端（命令式+符号式）设计和 Gluon API。

**Spark MLlib**：掌握 Spark 分布式机器学习库在大规模数据上的应用——Pipeline API（Transformer+Estimator 链式组合）、分布式训练（基于 RDD/DataFrame 的并行化）。

**DeepSpeed**：掌握微软大模型训练优化库的核心技术——ZeRO 三阶段（优化器状态/梯度/参数切分减少显存占用）、混合精度训练、梯度累积、ZeRO-Offload（将部分状态卸载到 CPU 内存/NVMe）；理解 DeepSpeed 在百亿级大模型训练中的关键作用。

**Transformers（HuggingFace）**：熟练掌握 HuggingFace Transformers 库——预训练模型加载（AutoModel/AutoTokenizer）、微调训练（Trainer API）、Pipeline 推理；熟悉模型库（Model Hub）生态和 Datasets 库在 NLP/CV 任务中的工程效率提升。

## 5. 开发工具

**集成开发环境**：熟练使用 VS Code、PyCharm、IntelliJ IDEA 等 IDE；掌握断点调试、远程开发（SSH Remote）、插件生态的高效利用。

**代码规范与检查工具**：熟悉华为代码规范要求；掌握静态分析工具（ESLint/Pylint/SonarQube）的配置与集成；理解代码规范在团队协作和代码质量保障中的作用。

**版本控制工具**：熟练掌握 Git 工作流——分支策略（Trunk-based/GitFlow）、合并与变基（merge vs rebase）、冲突解决、Cherry-pick、交互式 rebase；具备 GitHub/GitLab PR/MR 代码评审的实践经验。

**单元测试框架**：掌握单元测试编写规范——AAA 模式（Arrange-Act-Assert）、测试隔离（mock/stub 依赖）、边界条件覆盖；熟悉 pytest（Python）、JUnit（Java）、Jest（JavaScript）等测试框架的使用。

**持续集成与部署工具**：掌握 CI/CD 流水线设计——代码提交→自动构建→自动测试→自动部署的全流程；熟悉 Jenkins/GitHub Actions/GitLab CI 的配置与使用；理解 CI/CD 在保障代码质量和加速交付中的工程价值。

## 6. 功能测试

**单元测试**：掌握白盒测试方法——语句覆盖、分支覆盖、路径覆盖；理解单元测试作为质量保障第一道防线的重要性。

**压力测试**：掌握系统性能基准测试方法——使用 JMeter/Locust 等工具模拟并发请求，评估系统的吞吐量（QPS/TPS）、响应时间（P50/P99）、资源利用率；理解压力测试在发现性能瓶颈中的作用。

**自动化测试**：掌握测试自动化框架（Selenium/Playwright UI 自动化、API 自动化测试）的搭建与维护；理解自动化测试在回归测试效率提升中的价值。

**集成测试**：掌握模块间接口联调测试方法——验证组件间交互的正确性；理解集成测试与单元测试的粒度区别——关注组件协作而非单个函数。

**安全测试**：了解 OWASP Top 10 安全风险（SQL 注入、XSS、CSRF 等）的检测方法；熟悉安全扫描工具的使用。

**静态代码分析**：掌握在不运行程序的前提下通过源码分析发现潜在缺陷（空指针、资源泄漏、死代码）的方法；熟悉 SonarQube、Coverity 等工具的集成使用。

## 7. 我的学习感悟

软件基础这块是我觉得最"慢热"但最有长期价值的知识。

说实话，我大学学设计模式的时候基本是背书状态——工厂模式是什么、观察者模式是什么，考试能写出来就行。真正理解这些东西是在工作之后。有一次我接手一个代码量不大但扩展需求很多的项目，前任写的是一大堆 if-else，每加一个新功能就要改好几个地方。我用策略模式重构了一遍，之后再加新功能基本只需要新增一个类，不用动已有代码。那一刻我才真正理解了"开闭原则"不是空话。

关于 Python 的 GIL 问题，这是我在做数据预处理 pipeline 的时候踩的坑。当时用多线程去做 CPU 密集型的特征计算，结果发现速度反而比单线程还慢。查了资料才知道 GIL 的存在意味着 Python 多线程在 CPU 密集场景下几乎没有收益，改成多进程后速度立刻上去了。虽然是个基础知识，但不亲自踩一次坑真的记不住。

Git 工作流是另一个我觉得在团队协作中特别重要的东西。刚工作的时候我只会 add、commit、push 三连，遇到冲突就慌。后来逼着自己学了 rebase、cherry-pick、交互式 rebase 这些操作，代码管理的灵活度完全不一样了。尤其是 rebase 和 merge 的选择，我现在的习惯是个人分支用 rebase 保持线性历史，团队主分支用 merge 保留合并记录。

单元测试这件事，我以前觉得是"锦上添花"的，写不写都行。直到有一次改了一个公共函数的边界条件处理逻辑，自以为只影响自己的模块，结果上线后别人的模块出了问题。如果当时有完善的单元测试覆盖，CI 环节就能拦住。从那以后我对测试的态度彻底变了——不是"有时间再写"，而是"不写不放心"。

最后说一下 CI/CD。手动构建、手动部署的痛苦我是经历过的——忘记合并某个分支、漏跑了某个测试、部署的是错误版本……这些问题在配好 CI/CD 之后基本就消失了。我觉得 CI/CD 最大的价值不是自动化本身，而是它强迫你把构建和部署的每一步都标准化、可重复化，这对工程质量的提升是系统性的。
